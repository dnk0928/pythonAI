# Выпускной проект

## Задача

Разработать скрипт (или небольшой пакет), который **автоматизирует полезную сетевую задачу** в lab-среде и объединяет минимум **три технологии** из курса, например:

- **Netmiko** — SSH, сбор `show mac address-table` (или аналога) с **нескольких** коммутаторов;
- **Jinja2** — генерация отчёта в **HTML** или **Markdown**;
- **TextFSM или regex** — нормализация MAC/VLAN/портов в таблицу;
- **Работа с файлами** — инвентарь (JSON/YAML), лог ошибок подключения.

Допустимы замены по согласованию: NAPALM + YAML + JSON Schema; NETCONF + lxml + Jinja2 — главное, чтобы было **не менее трёх** осознанно используемых компонентов.

## Требования

1. **Инвентарь** вне кода: файл JSON или YAML из репозитория или путь аргументом CLI.
2. **Секреты** только через переменные окружения (например `LAB_SSH_USER`, `LAB_SSH_PASS`).
3. **Повторяемый запуск:** `python3 -m mac_report` или `python3 mac_report.py` с понятными аргументами (`--inventory`, `--output-dir`).
4. **Обработка ошибок:** сбой на одном устройстве не должен обязательно прерывать весь прогон; итог должен это отражать.
5. **Структура проекта:** минимум два модуля (например `inventory.py`, `collect.py`, `report.py`) + каталог `templates/`.
6. **README** в корне мини-проекта: как установить зависимости, как запустить, что ожидать на выходе.

## Критерии приёмки (чеклист)

- [ ] Три и более технологий из списка курса использованы по делу.
- [ ] Есть пример выходного отчёта в `artifacts/` или скриншот/описание формата.
- [ ] Код читаемый: функции с именами по смыслу, без магических чисел без констант.
- [ ] Зависимости зафиксированы в `requirements.txt` проекта.
- [ ] Описан **офлайн-режим** (чтение заранее сохранённых выводов), если нет доступа к lab.

## Идеи тем (выберите одну)

**Основные (этапы 1–3):**
- Сбор MAC-таблиц + отчёт по VLAN с пересечениями.
- Проверка «ожидаемых» описаний интерфейсов против шаблона.
- Сбор версий ПО + сверка с допустимым списком (JSON Schema).

**2026-ориентированные (с использованием Stage 4):**

- **Streaming telemetry + alert:** подписаться через gNMI (топик B4) на счётчики ошибок интерфейсов; при превышении порога записать предупреждение в лог и отправить HTTP-уведомление (httpx, топик B3) в webhook (например, Telegram или mock-сервер). Инвентарь — pydantic-модели (топик B2).

- **REST API + Jinja2-отчёт:** опросить Cisco DNA Center (или mock из `labs/mock_api/`) по всем устройствам через httpx; собрать reachability, версии, аптайм; сгенерировать Jinja2 HTML-отчёт. Параллельный опрос через AsyncClient (топик B1/B3).

- **ИИ-ассистент для анализа `show` вывода:** для каждого устройства из инвентаря выполнить `show interfaces` (Netmiko) или прочитать из `labs/sample_outputs/`; передать вывод в LLM API (топик B5) и попросить резюмировать проблемы (ошибки, down-интерфейсы); сохранить ответы в Markdown-отчёт. Все входные данные заменены заглушками — производственные данные в LLM не отправлять.
